Generative courses — what I'm going to build

Turn the static buildRoadmap() + three hand-authored lessonScripts into a real generation pipeline. The learner watches Lumen outline the path, then write the first lesson, then quietly fill in the rest and gather resources from the web — all as live, honest actions on screen, not a fake progress bar.

The three things this doc nails down
  • Deterministic UI: the model only emits params that fill Zod schemas; we own every React component. No model-authored HTML/JS ever reaches MathCanvas.
  • Modules-first, resources-after: lesson content streams first and unlocks fast; web research / sources / extra practice / concept animations stream in a second background wave — they never block reading.
  • Actions-driven animation: the existing onboarding "building your path" state becomes a live feed of the real actions the pipeline is taking (outlining → writing → filling in → gathering).

The two-wave streaming model the key idea

"Stream the modules first, then stream the resources in the background." Content is the foreground promise; resources are an enrichment that arrives after — never a gate in front of the first lesson.

The registry is the single source of truth both channels import from — no drift between "what the model can call" and "what we can render":

// src/lib/course-gen/ui-registry.ts — allowlist, imported by server tools AND client switch
export const GEN_UI = {
  showParabola:        { component: ParabolaWidget,        args: parabolaArgsSchema },
  showAlgebraTiles:    { component: AlgebraTiles,          args: tilesArgsSchema },
  showNumberLine:      { component: NumberLine,            args: numberLineArgsSchema },
  showPracticeCard:    { component: PracticeCard,          args: practiceArgsSchema },
  showWorkedExample:   { component: WorkedExample,         args: workedArgsSchema },
  showConceptAnimation:{ component: ConceptAnimationPlayer,args: conceptAnimationSchema },
  updateLessonDiagram: { component: null /* mutates script, no render */, args: diagramSchema },
} as const;
  • Allowlist only — a tool name not in GEN_UI renders nothing. No eval, no model HTML/CSS/JS.
  • Server owns arithmetic — enrichParabola recomputes roots/vertex from a,b,c before the client trusts them, so Live circles the right point.
  • LaTeX sanitized — length + charset cap, then katex.renderToString({throwOnError}) smoke test; a bad field is repaired or dropped, never streamed raw.

The modules pipeline, done well

One orchestrator, a priority queue, a validation gate, and failure isolation so a bad Module 3 can never take down Module 1.

  1. generateRoadmap — streamedstreamObject(roadmapSchema). Emits roadmap_partial per module for the pop-in. Resolves in < 5s to Course{ modules: [{status:"pending"}] }.
  2. generateOne(0) — priority, foregroundMistral streamObject(lessonScriptSchema). Partials stream as module_partial (skeleton only). Final → Zod → enrichDiagrams → math checks → ready.
  3. mapPool(modules[1…], 2–3) — background contentSame generator, no curation. Each flips generating → ready | failed, streamed as module_status / module_ready.
  4. resource pass — background, per ready moduleenrichModuleResources(): Mistral + Tavily search → citations, one extra practice, optional concept animation → patched onto the script → module_enriched.
  5. boost on openClient opens Module k → POST /api/course/boost bumps k+1's weight so "next lesson" is usually ready by the time they finish.
  6. failure isolationTwo failed gen attempts (small → large repair) → status:"failed" + retry on that card only. Soft fallback: inject a short placeholder script so the path never dead-ends.

The one SSE contract

POST /api/course/start → text/event-stream, served as a same-origin TanStack Start API route (src/routes/api/course/start.ts) — the pattern the app now uses for lumen-token and payments/*. No proxy, no separate process. Both the actions animation and the roadmap subscribe to the same events; GET /api/course/:id serves reconnect on a single-node server.

Architecture note (reconciled after merge): the repo was refactored to colocate server APIs — the old external token-server was removed and secrets now live in the app's server env, read via process.env inside createFileRoute(...).server.handlers. So generation lives in src/server/course/ (TypeScript, importing the shared @/lib/course-gen schemas + math — no JS duplication), the SSE route streams a ReadableStream, and the AI SDK installs in the frontend, never the browser bundle. The in-memory course Map is pinned to globalThis; a Cloudflare/edge deploy would swap it for KV/Durable Objects (the client also persists the course in localStorage). 
event: course          // { id, topic }
event: roadmap_partial // { modules: [...] }  ← pop-in animation
event: roadmap         // { modules: [{id,title,blurb,status:"pending"}] }  final outline
event: module_status   // { id, status:"generating" }
event: module_partial  // { id, partial }  ← skeleton only, never practice answers
event: module_ready    // { id, script }  ← Wave A content, board-ready
event: module_enriched // { id, resources }  ← Wave B, background
event: module_status   // { id, status:"failed", error }
event: done            // {}
  • Keys are server-only: MISTRAL_API_KEY, TAVILY_API_KEY — never VITE_-prefixed. Generation runs inside TanStack Start server routes.
  • Course persisted in an in-memory Map<courseId, Course> for the demo; SQLite/Turso later. Reconnect: GET /api/course/:id polls status; replay only missed readies.
  • SSE only — no second realtime stack. LiveKit (the Live tutor) stays independent.
